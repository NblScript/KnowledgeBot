# ADR-001: 采用微服务架构

## 状态

已接受

## 背景

KnowledgeBot 作为一个企业级 RAG 知识库问答系统，需要满足以下需求：

1. **可扩展性**: 支持企业级并发，部分服务（如嵌入服务、检索服务）需要独立扩展
2. **灵活性**: 支持多种 LLM 后端切换，支持定制化部署
3. **可维护性**: 团队协作开发，各模块独立部署更新
4. **多租户**: SaaS 模式部署，租户数据隔离

约束条件：
- 初期团队规模较小（3-5人）
- 需要快速迭代
- API 调用外部 AI 服务，非本地部署

## 决策

采用微服务架构，将系统拆分为以下独立服务：

### 核心服务
- **API Gateway**: 统一入口、路由、限流
- **Auth Service**: 认证授权、JWT 管理
- **User Service**: 用户管理、租户管理
- **Chat Service**: 对话编排、流式输出
- **LLM Service**: LLM 调用、模型管理

### 业务服务
- **Knowledge Base Service**: 知识库管理
- **Document Service**: 文档上传、解析、管理
- **Embedding Service**: 向量嵌入、异步处理
- **Retrieval Service**: 向量检索、重排序

## 后果

### 正面影响
- 各服务可独立部署和扩展
- 故障隔离，单服务故障不影响整体
- 技术栈灵活，各服务可选择最佳技术
- 团队可并行开发

### 负面影响
- 运维复杂度增加
- 服务间通信开销
- 分布式事务处理复杂
- 需要完善的服务治理

### 风险
- 初期开发成本较高
- 需要投入 DevOps 建设
- 服务拆分边界需要持续调整

## 替代方案

### 方案 A: 单体架构

单体架构，所有功能集成在一个应用中。

**优点**:
- 开发简单，部署方便
- 无服务间通信开销
- 事务处理简单

**缺点**:
- 扩展困难，无法按需扩展
- 故障影响全局
- 技术栈统一受限

**不选择的原因**: 无法满足不同服务独立扩展的需求，特别是嵌入服务和检索服务需要独立伸缩。

### 方案 B: 模块化单体

单体架构但模块化设计，保持良好的模块边界。

**优点**:
- 模块边界清晰，便于未来拆分
- 开发部署相对简单
- 避免过早微服务化

**缺点**:
- 模块间耦合风险
- 部署需要整体发布
- 无法独立扩展

**不选择的原因**: 嵌入服务需要独立部署和扩展，模块化单体无法满足。
